# RFC-0005: Randomness and Live Draw

## Status

Draft

## Summary

Randomness determines draw results and is therefore one of the highest-risk parts of TRT. The MVP should use a conservative oracle-based randomness approach on testnet before attempting live-draw or hybrid randomness.

This RFC defines randomness modes, minimum verification requirements, threat categories, and the recommended MVP path. It does not approve a production oracle or mainnet process.

Related preparation artifacts:

- [Oracle Randomness Payload](../contracts/ton/ORACLE_PAYLOAD.md)

## Motivation

Transparent draw infrastructure depends on users trusting that results are not manipulated. On-chain randomness is difficult, live draw inputs are operationally complex, and centralized oracles introduce trust assumptions.

TRT needs a phased randomness design that starts conservative, testable, and auditable, then evolves toward richer live-draw verification only after the basic protocol is secure.

## Specification

### Randomness goals

The randomness system should provide:

- unpredictability before entry lock;
- immutability after submission;
- public verifiability where practical;
- replay protection;
- domain separation per draw;
- clear operator/oracle accountability;
- deterministic result derivation by the draw contract.

### Randomness modes

#### Mode 1: Oracle-based randomness

An approved oracle service signs or submits a random value for a specific draw after entries are locked.

Minimum requirements:

- oracle identity is configured before draw creation or before entry lock;
- submission is accepted only after lock;
- submission includes draw-specific domain separation;
- value can be submitted only once;
- replayed payloads are rejected;
- signature/proof format is documented;
- oracle public key or authorized address is visible.

This is the recommended MVP mode for testnet.

#### Mode 2: Live draw input

Live draw numbers or physical process outputs are captured off-chain and submitted to the protocol.

This mode is not approved for MVP implementation. It requires separate operational design for:

- stream process;
- input capture;
- fraud prevention;
- multiple observers;
- evidence retention;
- dispute handling;
- oracle submission rules.

#### Mode 3: Hybrid randomness

Hybrid randomness combines multiple inputs, for example:

```text
finalRandom = hash(liveDrawNumbers + oracleRandom + drawDomain)
```

Block data may be considered only if manipulation risks and validator incentives are analyzed. Hybrid mode is not approved for MVP implementation.

### Domain separation

Randomness payloads must be bound to a specific draw. A future implementation should include a domain such as:

- protocol identifier;
- factory address;
- draw contract address;
- draw id or nonce;
- chain/network identifier;
- locked entry count;
- randomness mode;
- version.

Exact encoding must be defined before coding.

The current draft payload policy is tracked in [ORACLE_PAYLOAD.md](../contracts/ton/ORACLE_PAYLOAD.md). It is not approved as final binary serialization.

### Timing

Randomness must not be accepted before entry lock. The contract should define:

- earliest valid submission time;
- latest expected submission time;
- fallback if oracle does not submit;
- cancellation or refund path if randomness is unavailable.

Fallback behavior is not approved by this RFC and must be defined before implementation.

### Result derivation

Randomness is an input, not the full result. The draw contract must derive result data deterministically using:

- accepted randomness;
- locked entry data;
- draw parameters.

The winner/result algorithm must be specified in the draw contract RFC or implementation design before coding.

### MVP recommendation

The recommended MVP is:

- one oracle-based randomness mode;
- testnet only;
- one authorized oracle key/address;
- one-time submission after lock;
- deterministic result derivation;
- extensive tests for replay, wrong oracle, wrong draw id, early submission, and duplicate submission.

## Security Considerations

Randomness risks include:

- oracle bias;
- oracle withholding;
- oracle key compromise;
- payload replay;
- wrong draw binding;
- early submission before final entries;
- replacement after unfavorable result;
- last-entry manipulation;
- predictable seed generation;
- weak off-chain entropy;
- indexer displaying unverified results.

Required tests:

- reject randomness before lock;
- reject unauthorized oracle;
- reject duplicate randomness;
- reject randomness for a different draw;
- reject malformed payload;
- accept valid randomness once;
- deterministic result derivation from fixed inputs;
- behavior when randomness is missing, once fallback is specified.

### Oracle accountability

Oracle implementation must eventually document:

- key custody;
- signing process;
- entropy source;
- monitoring;
- incident response;
- key rotation;
- compromised key response.

## Legal / Compliance Considerations

Randomness affects fairness claims. User-facing language must not claim "provably fair" unless the proof model is accurate, documented, and reviewed.

Legal review should address:

- whether live-draw presentation creates additional regulatory obligations;
- whether oracle trust assumptions must be disclosed;
- whether disputes require formal procedures;
- whether result publication language is acceptable;
- whether evidence retention is required.

## Operational Considerations

Before public testnet:

- define oracle operator;
- define oracle key creation process;
- define signing/submission workflow;
- define monitoring for missing randomness;
- define test evidence retention.

Before live-draw mode:

- define broadcast process;
- define independent observation;
- define recording retention;
- define manual override prohibition or controls;
- define dispute process.

## Codex Implementation Notes

Codex must not implement randomness submission or winner selection until this RFC and the draw contract algorithm are approved.

The next engineering step may be a randomness test matrix and oracle interface design. Initial implementation should be testnet-only and use a conservative signing scheme.

Before implementation, Codex should verify the RFC status using [RFC-PROCESS.md](RFC-PROCESS.md).

## Claude Code Review Notes

Claude Code should verify:

- randomness is bound to a specific draw;
- oracle trust assumptions are explicit;
- duplicate and replay submissions are rejected;
- live-draw and hybrid modes are not implemented prematurely;
- "provably fair" claims are not made without a real proof model;
- tests cover manipulation and timing edge cases.
