# RFC-0003: TON Draw Factory

## Status

Approved

## Summary

The TON Draw Factory is the protocol component responsible for creating individual draw contracts with explicit parameters, deterministic metadata, and indexable deployment events.

The factory must not contain winner-selection logic or payout logic. Its job is to validate draw creation parameters, deploy a draw contract, record enough data for off-chain indexing, and enforce creator/admin boundaries.

This RFC defines the intended factory responsibilities and constraints. It does not approve mainnet deployment or public paid draws.

## Approved Scope

This RFC is approved for the Sprint 4 Factory MVP only. It authorizes implementation of the factory boundary described below for local/testnet development with free-entry draws only.

Approval of this RFC must not be interpreted as approval for:

- participant entry logic;
- draw locking;
- randomness submission;
- winner/result derivation;
- claims, refunds, or payouts;
- creator fees or protocol fees;
- TON paid entries;
- Jetton paid entries;
- project token mechanics;
- charity allocation or automation;
- creator self-serve public launch;
- mainnet deployment.

RFC-0004 and RFC-0005 must be reviewed for compatibility before Sprint 4 implementation starts, but they are not approved by this RFC.

## Motivation

A factory gives TRT a consistent way to create draw contracts while keeping each draw isolated. Isolated draw contracts are easier to audit, test, index, and reason about than one large contract managing all draws.

The factory also creates a clear boundary between creator configuration and protocol rules. This boundary is important because creator-controlled parameters can introduce legal, economic, and security risks.

## Specification

### Terminology

Technical documents should prefer "draw" and "entry". Existing file names may still use "lottery" for continuity, but new implementation names should favor conservative terminology unless legal review approves otherwise.

### Responsibilities

The factory is responsible for:

- accepting draw creation requests;
- validating supported draw parameters;
- deploying one independent draw contract per draw;
- emitting or exposing indexable deployment metadata;
- preserving creator ownership boundaries;
- applying protocol-level creation constraints;
- optionally collecting a creation fee after separate approval;
- exposing factory configuration for frontend and indexer use.

The factory is not responsible for:

- accepting participant entries;
- locking entries;
- generating randomness;
- selecting winners;
- calculating payouts;
- distributing funds;
- managing charity operations;
- launching tokens;
- making legal eligibility decisions by itself.

### Draw creation parameters

A future implementation may support the following parameter categories after approval:

- creator address;
- draw title or metadata reference;
- entry mode: free-entry, TON payment, Jetton payment, or other approved mode;
- entry price where applicable;
- maximum entries;
- entries per participant limit;
- start time;
- entry lock time;
- draw finalization time;
- reward or prize configuration;
- creator fee;
- protocol fee;
- charity allocation;
- randomness mode;
- oracle public key or approved oracle identifier;
- cancellation rules;
- claim deadline;
- metadata URI or content hash.

Not all parameters must be implemented in the MVP. Each implementation sprint must explicitly list the supported subset.

### Validation requirements

The factory must reject unsupported or unsafe configurations, including:

- zero or invalid creator address;
- unsupported payment asset;
- negative or overflow-prone fee values;
- total fees above approved caps;
- draw times in invalid order;
- randomness mode without required oracle configuration;
- metadata formats not supported by the current indexer/frontend;
- configurations that exceed storage or gas assumptions.

Specific numeric caps must be defined in the implementation RFC or technical design before coding.

### Deployment model

The preferred model is deterministic deployment where TON tooling and contract design make it practical. If deterministic deployment is not used, the RFC must explain why and define an indexing strategy.

Each draw contract should be initialized with immutable parameters where possible. Mutable parameters must be minimized and explicitly documented.

### Events and indexing

The factory must expose enough information for backend indexers and public verification tools to discover:

- factory version;
- draw contract address;
- creator address;
- creation timestamp or logical time;
- supported parameter hash;
- metadata reference;
- randomness mode;
- payment asset;
- fee configuration.

Exact event/message formats must be defined during implementation design.

### Admin controls

Factory admin controls should be minimal. Potential controls may include:

- setting protocol fee caps;
- setting supported assets;
- setting approved randomness modes;
- pausing new draw creation;
- updating metadata about factory version.

Admin controls must not be able to change already deployed draw parameters unless a later RFC explicitly approves an upgrade or migration mechanism.

### Sprint 4 Factory MVP scope

The reviewed Sprint 4 Factory MVP scope is:

- expose factory version state;
- expose factory admin address;
- support a paused/unpaused creation flag;
- support one oracle-based randomness mode;
- accept creator address;
- accept metadata URI or content hash;
- accept start time, entry lock time, and draw finalization time;
- accept maximum entries;
- accept maximum entries per participant;
- accept free-entry mode only;
- accept oracle identity for the approved oracle-based randomness mode;
- accept protocol version;
- deploy or prepare independent local/testnet draw contracts;
- expose created draw address or init data for wrappers/indexers.

The Sprint 4 Factory MVP must reject:

- invalid creator address;
- invalid time ordering;
- zero maximum entries;
- zero participant entry limit;
- participant entry limit greater than maximum entries;
- unsupported randomness mode;
- missing oracle identity;
- unsupported metadata format;
- creation while paused.

This Approved status authorizes implementation only inside the Sprint 4 Factory MVP scope above.

The current first-MVP subset is tracked in [MVP_SCOPE.md](../contracts/ton/MVP_SCOPE.md). If this RFC is approved for Sprint 4, implementation must stay within that subset unless the RFC is updated.

### Required Sprint 4 tests

The approved Sprint 4 implementation must include tests for:

1. Factory exposes protocol version.
2. Admin can pause and unpause creation.
3. Non-admin cannot pause or unpause creation.
4. Factory rejects invalid creator address.
5. Factory rejects invalid time ordering.
6. Factory rejects zero maximum entries.
7. Factory rejects zero participant entry limit.
8. Factory rejects participant limit greater than maximum entries.
9. Factory rejects unsupported randomness mode.
10. Factory rejects missing oracle identity.
11. Factory accepts valid MVP draw parameters.
12. Factory exposes created draw address or init data.

## Security Considerations

The main factory risks are:

- unsafe parameter creation;
- fee misconfiguration;
- creator/admin privilege confusion;
- non-indexable draw deployments;
- replayed or spoofed creation messages;
- deploying contracts with invalid code or init data;
- storage/gas assumptions that fail at scale.

Factory implementation must include tests for:

- accepted valid creation parameters;
- rejected invalid time ordering;
- rejected unsupported payment assets;
- rejected invalid fee configuration;
- rejected missing randomness configuration;
- correct draw address exposure;
- admin-only configuration changes;
- non-admin rejection paths.

## Legal / Compliance Considerations

The factory can make it easy for creators to deploy legally sensitive draw mechanics. Therefore, creator self-service must remain disabled or restricted until legal review approves the operating model.

Legal review should answer:

- who is the operator of creator-created draws;
- whether creators need jurisdiction restrictions;
- whether paid entries require KYC/AML or age gates;
- whether free-entry mode changes legal classification;
- whether creator fees or protocol fees are permitted;
- whether charity allocations can be configured by creators;
- what user-facing terminology is allowed.

## Operational Considerations

Before public testnet usage, the project should define:

- factory deployment owner;
- testnet upgrade/redeploy process;
- indexer discovery process;
- versioning strategy;
- rollback plan for broken factory releases;
- draw creation monitoring.

Mainnet requires a separate launch checklist and independent audit.

## Codex Implementation Notes

Codex may implement factory business logic only inside the approved Sprint 4 Factory MVP scope. Any draw entry, randomness submission, result derivation, payout, fee, token, charity, creator self-service, or mainnet behavior still requires a separate approved RFC.

When approved, implementation should likely use Tact with TypeScript tests through TON Sandbox/Blueprint.

Before implementation, Codex should verify the RFC status using [RFC-PROCESS.md](RFC-PROCESS.md).

## Claude Code Review Notes

Claude Code should verify:

- factory scope does not include draw winner or payout logic;
- creator/admin boundaries are clear;
- parameter validation is complete enough for MVP;
- indexing needs are explicitly addressed;
- legal-risk features are gated;
- future implementation does not exceed approved RFC scope.
