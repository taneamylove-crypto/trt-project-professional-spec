# RFC-0004: TON Draw Contract

## Status

Draft

## Summary

Each TRT draw should be represented by an independent TON smart contract deployed by the factory. The draw contract owns the lifecycle of one draw: accepting entries, locking entries, accepting approved randomness, deriving result data, and exposing claim or settlement state.

This RFC defines the target contract responsibilities, state machine, invariants, and review requirements. It does not approve implementation yet.

Related preparation artifacts:

- [TON Contract MVP Scope](../contracts/ton/MVP_SCOPE.md)

## Motivation

An independent contract per draw provides isolation. A bug, cancellation, or operational issue in one draw should not corrupt unrelated draws. It also improves transparency because participants and reviewers can inspect one contract's state, parameters, entries, randomness, and finalization status.

The draw contract is the most security-sensitive part of TRT because it will eventually protect user funds, entry accounting, randomness acceptance, and reward claims.

## Specification

### Responsibilities

The draw contract is responsible for:

- storing immutable draw parameters;
- accepting valid entries during the entry window;
- recording entry ownership or participant accounting;
- enforcing entry limits;
- locking entries before finalization;
- accepting randomness only from the approved source;
- calculating deterministic result data from locked entries and randomness;
- exposing final state for indexers and public verification;
- supporting claim or settlement mechanics after approval.

The draw contract is not responsible for:

- deploying other draws;
- changing protocol-wide factory settings;
- performing off-chain legal eligibility checks;
- making medical, charity, or token claims;
- generating randomness internally without an approved randomness RFC;
- handling creator-dashboard workflows.

### State machine

The target state machine is:

1. `Pending`: contract deployed, not yet open for entries.
2. `Open`: entries are accepted.
3. `Locked`: entry list/accounting is fixed.
4. `RandomnessSubmitted`: approved randomness has been accepted.
5. `Finalized`: result data is derived and exposed.
6. `Claiming`: eligible claims or settlements are available, if applicable.
7. `Closed`: draw is complete or claim deadline passed.
8. `Cancelled`: draw is cancelled under approved cancellation rules.

The MVP may merge or omit states only if the implementation RFC explains why and preserves security invariants.

### Entry model

The preferred conceptual model is "entries", not "tickets". The implementation may store entries as:

- one record per entry;
- participant address plus entry count;
- compressed ranges;
- another auditable structure.

The implementation RFC must justify the storage model based on TON gas/storage constraints, result verification needs, and frontend/indexer requirements.

### Entry acceptance rules

The contract must reject entries when:

- the draw is not open;
- the sender or payment does not match the configured entry mode;
- the entry amount is invalid;
- participant entry limits would be exceeded;
- maximum entries would be exceeded;
- payment asset is unsupported;
- the draw has been locked, finalized, closed, or cancelled.

### Locking

Entry locking freezes the participant accounting used for result calculation. After locking:

- no new entries may be accepted;
- entry ownership/accounting must not change;
- the final entry count must be visible or derivable;
- randomness acceptance may become available according to the configured mode.

### Randomness acceptance

The contract must accept randomness only under RFC-0005 rules. At minimum, it must verify:

- authorized randomness submitter;
- expected draw identifier or domain separation;
- timing constraints;
- one-time submission;
- replay protection;
- payload shape and proof/signature where required.

### Result calculation

Winner/result derivation must be deterministic and publicly reproducible from:

- locked entry accounting;
- accepted randomness;
- draw parameters.

Exact calculation must be defined before implementation. This RFC does not approve a specific algorithm.

The current draft result policy is tracked in [RESULT_DERIVATION.md](../contracts/ton/RESULT_DERIVATION.md). It is not approved as final winner-selection implementation.

### Payout and claim model

Payout mechanics are not approved by this RFC. A later implementation RFC must define:

- reward pool custody;
- fee deductions;
- winner claim rules;
- unclaimed funds handling;
- cancellation refunds;
- failed transfer behavior;
- rounding behavior;
- gas funding requirements.

Until then, implementation may only expose placeholder or test-only state.

### Cancellation

Cancellation is legally and technically sensitive. If supported, cancellation rules must define:

- who can cancel;
- which states allow cancellation;
- participant refund behavior;
- creator/protocol fee behavior;
- event/indexer visibility;
- abuse mitigation.

### Required invariants

Every implementation must preserve:

- entries cannot be added after lock;
- randomness cannot be accepted before lock;
- randomness cannot be replaced after acceptance;
- result cannot be finalized without accepted randomness;
- non-admins cannot perform admin actions;
- funds cannot be claimed by unauthorized addresses;
- final result calculation is deterministic;
- cancelled draws cannot finalize;
- closed draws cannot mutate security-critical state.

## Security Considerations

Critical risks include:

- randomness manipulation;
- last-entry attacks;
- entry accounting corruption;
- payout theft or trapped funds;
- admin abuse;
- replayed randomness;
- gas griefing;
- denial of finalization;
- incorrect handling of Jettons or TON transfers;
- off-by-one winner selection;
- state-machine bypasses.

Required test categories:

- state transition tests;
- entry acceptance/rejection tests;
- lock/finalize tests;
- randomness authorization tests;
- result determinism tests;
- claim authorization tests when claims are implemented;
- cancellation/refund tests when cancellation is implemented;
- gas/storage budget tests.

## Legal / Compliance Considerations

The draw contract must not be treated as legally neutral merely because it is on-chain. The combination of consideration, chance, and prize may create gambling or lottery risk.

Legal review must address:

- whether paid entries are allowed;
- whether free-entry alternatives are required;
- jurisdiction exclusions;
- age restrictions;
- creator/operator responsibility;
- refund obligations;
- prize disclosure requirements;
- tax reporting obligations.

## Operational Considerations

Before testnet deployment, operations should define:

- deployment process;
- testnet faucet/funding assumptions;
- contract versioning;
- indexer support;
- post-draw verification report;
- incident handling for stuck draws;
- contract source verification process.

Mainnet requires independent audit and launch approval.

## Codex Implementation Notes

Codex must not implement draw contract business logic until this RFC or a specific implementation RFC is approved.

The next engineering step may be a test matrix and Tact project design for the contract package. Any code implementation must begin with tests for the state machine and invariants.

Before implementation, Codex should verify the RFC status using [RFC-PROCESS.md](RFC-PROCESS.md).

## Claude Code Review Notes

Claude Code should verify:

- state transitions are complete and safe;
- invariants are testable;
- payout and randomness logic are not smuggled into scope prematurely;
- admin controls are minimized;
- result calculation is deterministic and reviewable once specified;
- tests cover edge cases before mainnet-related work begins.
