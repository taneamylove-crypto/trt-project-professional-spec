# RFC-0001: Vision and Scope

## Status

Draft

## Summary

TRT is a TON-native ecosystem project with a strict separation between three pillars:

1. Draw / Lottery Protocol
2. Charity Foundation
3. Mental Aid Institute

The first implementation focus is the Draw / Lottery Protocol: transparent draw infrastructure for TON and Telegram. The protocol should allow audited, understandable, and verifiable draw mechanics without mixing the protocol with charity operations, mental-aid activities, token promotion, or unapproved legal claims.

This RFC defines the project boundaries that all later RFCs and implementation work must respect.

## Motivation

TRT combines multiple sensitive domains: blockchain settlement, random winner selection, prize or reward distribution, charity funding, tokenomics, and psychological support. Without strong boundaries, the project can become legally confusing, technically unsafe, or impossible to review.

The project needs a foundation document before implementation so that future RFCs, code, reviews, and GitHub workflows all optimize for the same goal: a professional, auditable, RFC-driven product.

## Specification

### Product pillars

TRT contains three independent pillars.

The **Draw / Lottery Protocol** is the technical protocol layer. It may include TON smart contracts, a factory for draw contracts, backend indexing, Telegram Mini App interfaces, a creator dashboard, SDKs, and public verification tools.

The **Charity Foundation** is an independent social-impact structure. It may receive donations, grants, sponsorships, or protocol treasury allocations only through clearly documented and legally reviewed mechanisms.

The **Mental Aid Institute** is an independent education, support, and research structure. Evidence-based psychological support must remain clearly separated from spiritual, consciousness, or parapsychological research topics.

### Primary MVP scope

The first technical phase may prepare infrastructure for:

- TON smart-contract development;
- Telegram Mini App shell;
- backend service shell;
- SDK shell;
- Creator Dashboard shell;
- RFC, review, and CI workflow.

The first technical phase must not implement production draw business logic until the relevant RFCs are approved.

### Out of scope for early implementation

The following are explicitly out of scope until separate RFCs and reviews approve them:

- mainnet deployment;
- paid public lottery or gambling operations;
- token launch;
- staking, yield, or investment-style mechanics;
- medical, therapeutic, crisis-care, or treatment claims;
- automated prize payout logic;
- live draw oracle submission;
- creator self-serve launch flows;
- admin emergency controls beyond documented placeholders.

### RFC-first rule

Every major feature must follow this sequence:

1. Create or update an RFC.
2. Review product, technical, legal, security, and operational consequences.
3. Move the RFC to an approved status.
4. Implement only the approved scope.
5. Add tests and documentation.
6. Submit implementation for review.

### Terminology

Until legal review approves public language, technical documentation should prefer:

- "draw" over "lottery" where possible;
- "entry" over "ticket";
- "participant" over "player";
- "allocation" or "reward" over "winnings" where context allows.

Existing filenames may still use `LOTTERY` for continuity, but user-facing terminology should be conservative.

### Repository authority

This repository is the source of truth for:

- specifications;
- RFCs;
- implementation tasks;
- monorepo structure;
- CI checks;
- review history;
- future source code.

GitHub `main` should represent the current accepted baseline. `develop` should track integration work that is ready for shared review.

## Security Considerations

TRT must treat the following as high-risk areas:

- randomness generation and submission;
- entry locking and finalization;
- winner calculation;
- payout calculation and claim logic;
- admin permissions;
- treasury allocation;
- contract upgrade or pause mechanisms;
- oracle trust assumptions;
- frontend wallet signing UX;
- backend indexing correctness.

No mainnet deployment should happen before:

- unit tests;
- integration tests;
- threat model;
- internal review;
- independent TON smart-contract audit;
- oracle/randomness review;
- admin-key review;
- multisig setup where applicable.

## Legal / Compliance Considerations

This repository does not provide legal advice.

Before any public paid draw, token launch, charity allocation claim, or mental-aid service claim, TRT requires jurisdiction-specific legal review.

Legal review must cover at least:

- gambling / lottery regulation;
- crypto-asset regulation;
- consumer protection;
- AML/KYC;
- charity law;
- psychological service regulation;
- tax treatment;
- advertising and prize-claim language;
- Telegram Mini App distribution constraints.

Using TON, Telegram, token mechanics, "entries", or blockchain transparency does not automatically remove gambling-law risk.

## Operational Considerations

Early operations should prioritize testnet, documentation, and review. Any live draw process must eventually define:

- pre-draw announcement;
- entry lock;
- randomness generation;
- result submission;
- contract verification;
- payout or claim flow;
- post-draw report;
- incident response.

Production operations must not depend on a single private key or undocumented manual process.

## Codex Implementation Notes

Codex should:

- maintain monorepo structure and CI;
- prepare implementation shells without business logic;
- create RFCs before implementing major features;
- implement only approved RFC scope;
- keep protocol, foundation, and institute concerns separated;
- create separate commits for meaningful changes;
- keep GitHub synchronized with the local repository.

Codex should not:

- implement draw, payout, randomness, token, or charity transfer business logic from this RFC alone;
- deploy to mainnet;
- add public claims that bypass legal review;
- collapse the three project pillars into one operational entity.

## Claude Code Review Notes

Claude Code should verify:

- later RFCs remain consistent with this scope;
- implementation PRs do not add business logic without approved RFCs;
- terminology remains conservative where legal risk exists;
- security-critical assumptions are documented;
- protocol, foundation, and institute responsibilities remain separated.
