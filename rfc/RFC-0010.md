# RFC-0010: Legal and Security Boundaries

## Status

Draft

## Summary

This RFC defines non-negotiable legal and security boundaries for TRT. It does not approve any specific draw mechanics, tokenomics, randomness method, payout flow, charity structure, or mental-aid service. Instead, it establishes the review gates that must be satisfied before those areas can move from concept to implementation or launch.

TRT must be developed as a compliance-aware and security-first project, especially because it touches prize mechanics, crypto assets, public users, Telegram distribution, charity funding, and psychological support.

## Motivation

The highest-risk failure mode for TRT is not only a software bug. It is a mismatch between what the system does, what users believe it does, what regulators may classify it as, and what the smart contracts actually enforce.

This RFC exists to prevent avoidable ambiguity before implementation begins.

## Specification

### Boundary 1: Legal review before public launch

No public paid draw, token launch, staking mechanic, charity allocation claim, or mental-aid service claim may launch until legal review is complete for the target jurisdictions.

Legal review must be documented in `docs/80_legal/` or referenced there when the material is confidential.

### Boundary 2: No disguised lottery mechanics

TRT must not use token purchase, staking, NFT ownership, donation, or creator subscription as an indirect prize-chance mechanism unless legal review explicitly approves the structure.

If a user pays or gives value primarily to receive a random chance at a prize, the mechanism must be treated as legally sensitive regardless of terminology.

### Boundary 3: Protocol, operator, foundation, and institute separation

The protocol must not imply that:

- the protocol developer is automatically the lottery operator;
- the charity foundation is the same entity as the protocol operator;
- the Mental Aid Institute provides regulated medical services without licensing;
- token holders are entitled to charity funds or prize pools;
- protocol fees are automatically charitable donations unless the structure is approved.

### Boundary 4: Security review before testnet and mainnet

Testnet deployment requires:

- approved protocol RFCs;
- unit tests for contract logic;
- integration tests for expected flows;
- documented admin roles;
- documented randomness assumptions;
- documented failure modes.

Mainnet deployment additionally requires:

- independent TON smart-contract audit;
- threat model;
- oracle/randomness review;
- admin-key and multisig review;
- incident response plan;
- launch checklist.

### Boundary 5: Randomness cannot be informal

Randomness must be specified before implementation. A random value is not acceptable unless the RFC defines:

- source of randomness;
- who can submit it;
- verification method;
- manipulation risks;
- timing constraints;
- replay protection;
- fallback behavior;
- public proof or audit trail.

### Boundary 6: Admin power must be minimized

Admin controls must be explicitly documented. If an admin can pause, cancel, upgrade, replace randomness, change fees, redirect funds, or affect winner selection, the RFC must explain why the control exists and how abuse is mitigated.

### Boundary 7: User-facing claims must be conservative

Before legal approval, public language should avoid:

- guaranteed returns;
- investment framing;
- medical or therapeutic claims;
- "risk-free" prize language;
- charity impact claims without accounting support;
- claims that blockchain usage alone guarantees fairness.

### Boundary 8: No mainnet by default

All implementation work is testnet/local by default. Mainnet requires an explicit launch RFC or launch checklist approved after review.

## Security Considerations

Security review must focus on at least these threat categories:

- randomness manipulation;
- last-entry or timing attacks;
- entry list mutation after lock;
- duplicate entry accounting;
- payout underflow, overflow, or rounding errors;
- trapped funds;
- unauthorized claims;
- admin-key compromise;
- oracle-key compromise;
- frontend transaction spoofing;
- backend indexer inconsistencies;
- denial of service around draw finalization;
- replayed or stale oracle submissions.

Every smart-contract RFC must include:

- state machine;
- accepted messages;
- rejected messages;
- invariant list;
- test matrix;
- known limitations.

## Legal / Compliance Considerations

This repository does not provide legal advice.

Professional legal review should answer at least:

- Which jurisdictions are targeted or excluded?
- Is the draw mechanism gambling, lottery, sweepstakes, contest, promotional game, or another category?
- Does free-entry mode change the classification?
- Are KYC/AML controls required?
- Are minors excluded and how?
- Are Telegram Mini App distribution rules satisfied?
- Can creators legally operate their own draws?
- Can protocol fees fund a charity foundation?
- Can a token exist without being treated as prize-chance consideration?
- What language is allowed in marketing, UI, and documentation?
- What tax reporting duties may apply?
- What licenses or professional qualifications are required for mental-aid services?

Legal conclusions should be linked from this RFC or from `docs/80_legal/LEGAL_ARCHITECTURE.md` when they become available.

## Operational Considerations

Before any public test, TRT should have:

- environment separation: local, testnet, staging, production;
- key-management rules;
- deployer account documentation;
- monitoring and alerting plan;
- incident response owner;
- post-draw reporting template;
- rollback or pause policy where applicable;
- user support escalation route.

Before mainnet, TRT should also define:

- multisig participants;
- release approval process;
- audit remediation checklist;
- emergency communication channel;
- contract verification process;
- treasury reporting process.

## Codex Implementation Notes

Codex may implement:

- repository controls;
- CI checks;
- RFC templates;
- placeholder packages;
- local/testnet tooling after relevant RFC approval;
- tests for approved behavior.

Codex must not implement without approved RFCs:

- winner selection;
- payout logic;
- randomness submission;
- creator-created public draws;
- token mechanics;
- treasury allocation;
- charity transfer automation;
- mental-aid service workflows;
- mainnet deployment scripts.

When implementing approved security-sensitive code, Codex must include tests for success paths, rejected messages, permissions, edge cases, and failure modes.

## Claude Code Review Notes

Claude Code should review:

- whether later RFCs satisfy these boundaries;
- whether implementation PRs exceed approved scope;
- whether randomness and payout logic have manipulation risks;
- whether admin powers are overbroad;
- whether legal-risk terminology leaks into user-facing surfaces;
- whether tests cover invariants and edge cases;
- whether mainnet or production assumptions appear prematurely.
